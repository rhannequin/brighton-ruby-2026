Lie #1: A day is 24 hours

TODO: Egyptian sundial illustration

In ancient Egypt, around 2100 BCE, the day had twelve hours and the night had twelve hours. But the hours weren't the same length as ours. They stretched and shrank with the seasons. A summer day-hour was longer than a winter day-hour, because the sun was up longer and they still divided that span into twelve. And that was fine.

Equal-length hours: ~150 BCE
Hipparchus

It took the Greeks to give us the regular hour. We invented it. It is not a fact of nature. The hour is a human convention that's about two thousand years old.

Hours are a convention. Days are too.

Which is a problem if you're storing time as numbers. 


          Time.zone = "Europe/London"
          t = Time.zone.parse("2026-10-25 00:30:00")
          # => Sun, 25 Oct 2026 00:30:00 BST +01:00
          

          t + 86400
          # => Sun, 25 Oct 2026 23:30:00 GMT +00:00

On October 25, 2026, in London, a day is 90,000 seconds long. 


          t + 1.day
          # => Mon, 26 Oct 2026 00:30:00 GMT +00:00

1.day isn't a number. It's an ActiveSupport::Duration, a value object that knows what it means. ActiveSupport patches Time#+ to call advance instead of doing seconds. The fix is one character. Always reach for the object that means what you want, not the number you assume it equals.

Lie #2: The calendar is stable

Both are correct. Britain switched calendars in 1752 and just deleted eleven days. Newton's birth didn't move; the dates did. The same person was born on two days, depending on which country's calendar you ask.

Calendars change.

Time zones change.


      require "tzinfo"

      tz = TZInfo::Timezone.get("America/Asuncion")
      winter_utc = Time.utc(2025, 7, 1, 12, 0, 0)
      local = tz.utc_to_local(winter_utc)

      puts "UTC time  : #{winter_utc}"
      puts "Local time: #{local}"

      # UTC time  : 2025-07-01 12:00:00 UTC
      # Local time: 2025-07-01 08:00:00

Let's look at Paraguay. If you run this code on an old app that hasn't been updated for a while, it will give you a local time of 8 AM. 


      require "tzinfo"

      tz = TZInfo::Timezone.get("America/Asuncion")
      winter_utc = Time.utc(2025, 7, 1, 12, 0, 0)
      local = tz.utc_to_local(winter_utc)

      puts "UTC time  : #{winter_utc}"
      puts "Local time: #{local}"

      # UTC time  : 2025-07-01 12:00:00 UTC
      # Local time: 2025-07-01 09:00:00

Then, run the same on a recent version of Rails and you will see the same code produces a different time.

That's because Paraguay observed DST until late 2024, and then abolished it in 2025.

If your bundle is stale, your application is doing the right thing for a country that no longer exists. 


            $ bundle update tzinfo

Lie #3: Time zones are geography

Before the end of the 19th century, every town set its own clock by the sun. Noon in Bristol was not noon in London, because solar time is local. This was fine, because nobody could move faster than the Sun did. Then we invented trains. People started experiencing two local times in one day, and the railway timetables became unreadable.

So, some countries adopted a single standard time for the whole country, and the world was divided into time zones. Super easy, you just have to separate the world into 24 time zones, each one hour apart, and assign a standard time to each zone.

TODO: Picture of time zones map

But of course we couldn't do something that simple. Countries want to have a unified time for their people within their borders, not just based on geography. So we have time zones that are a mix between meridians and political borders.

Each time zone supports a local mean solar time that is applied to everyone within its borders.

IANA database: 600+ time zones

The IANA database stores more than 600 time zones and is updated several times a year to reflect changes in time zone rules, because somewhere in the world, a government has decided to start or stop doing something with their clocks.

Lie #4: `Time.now` gives current time


            config.time_zone = "UTC"

            Time.now      # => 2026-03-25 17:25:55 +0100
            Time.current  # => 2026-03-25 16:25:58 UTC

My configuration said UTC. Time.now said no. Time.now reads the system clock and the system zone. It does not consult your application. The fix is one word.

Time.current

Always.

Lie #5: `Time.parse` is reliable


            Time.parse("2026-03-29 09:00:00")
            # => 2026-03-29 09:00:00 +0200

There is no +0200 in the input. The +0200 came from my laptop. If I deploy this code to a server in another zone, it returns a different value for the same string. This is a very common bug and the fix is quite easy as well. 


            Time.zone.parse("2026-03-29 09:00:00")
            # => 2026-03-29 09:00:00 UTC

Five characters and your code won't assume anything on your behalf.
