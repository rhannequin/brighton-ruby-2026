Lie #1: A day is 24 hours


          Time.zone = "Europe/London"
          t = Time.zone.parse("2026-10-25 00:30:00")
          # => Sun, 25 Oct 2026 00:30:00 BST +01:00

          # => Sun, 25 Oct 2026 23:30:00 GMT +00:00

On October 25, 2026, in London, a day is 90,000 seconds long.

Truth #1


          t + 1.day
          # => Mon, 26 Oct 2026 00:30:00 GMT +00:00
  • 1.day isn't a number, it's an ActiveSupport::Duration
  • Value object that knows what it means
  • ActiveSupport patches Time#+ to call advance instead of doing seconds
  • The fix is one character

Use the object that means what you want.

  • This is the rule. The talk in one sentence.
  • 1.day is just the start — it generalises

          jan_31 = Time.zone.local(2026, 1, 31)

          jan_31 + 1.month   # => Sat, 28 Feb 2026
          jan_31 + 2.months  # => Tue, 31 Mar 2026
          jan_31 + 30.days   # => Mon, 02 Mar 2026
  • Three operations, three different answers
  • + 1.month → end-of-month aware (Feb 31 doesn't exist)
  • + 2.months → anchors to original day, NOT 1.month + 1.month
  • + 30.days → pure day count, ignores month boundaries
  • Under the hood, 1.month calls advance — same lesson, named units
  • Each unit has its own semantics. Pick the one that matches intent.

Same day, different hour


          Time.zone.now.change(
            hour: 9, min: 0, sec: 0
          )
  • change rewrites parts of a time without touching the rest
  • Useful for "9 AM today in the user's zone"
  • We'll see this again in the scheduling example

Lie #3: Your zone data is current

Calendars changed centuries ago. Zone rules are changing in our lifetime. Same shape of problem, just faster. 


      require "tzinfo"

      tz = TZInfo::Timezone.get("America/Asuncion")
      winter_utc = Time.utc(2025, 7, 1, 12, 0, 0)
      local = tz.utc_to_local(winter_utc)

      puts "UTC time  : #{winter_utc}"
      puts "Local time: #{local}"

      # UTC time  : 2025-07-01 12:00:00 UTC
      # Local time: 2025-07-01 08:00:00

Let's look at Paraguay. If you run this code on an old app that hasn't been updated for a while, it will give you a local time of 8 AM. 


      require "tzinfo"

      tz = TZInfo::Timezone.get("America/Asuncion")
      winter_utc = Time.utc(2025, 7, 1, 12, 0, 0)
      local = tz.utc_to_local(winter_utc)

      puts "UTC time  : #{winter_utc}"
      puts "Local time: #{local}"

      # UTC time  : 2025-07-01 12:00:00 UTC
      # Local time: 2025-07-01 09:00:00

Then, run the same on a recent version of Rails and you will see the same code produces a different time.

That's because Paraguay observed DST until late 2024, and then abolished it in 2025.

If your bundle is stale, your application is doing the right thing for a country that no longer exists.

Lie #4: Time zones are geography

Before the end of the 19th century, every town set its own clock by the sun. Noon in Bristol was not noon in London, because solar time is local. This was fine, because nobody could move faster than the Sun did. Then we invented trains. People started experiencing two local times in one day, and the railway timetables became unreadable.

TODO: Picture of meridians

So, some countries adopted a single standard time for the whole country, and the world was divided into time zones. Super easy, you just have to separate the world into 24 time zones, each one hour apart, and assign a standard time to each zone.

TODO: Picture of time zones map

But of course we couldn't do something that simple. Countries want to have a unified time for their people within their borders, not just based on geography. So we have time zones that are a mix between meridians and political borders.

Each time zone supports a local mean solar time that is applied to everyone within its borders.

IANA database: 600+ time zones

The IANA database stores more than 600 time zones and is updated several times a year to reflect changes in time zone rules, because somewhere in the world, a government has decided to start or stop doing something with their clocks.

Truth #4

Store IANA names.

Avoid Rails names

If you need more context than just a point in time, you might want to store not only the time but also the time zones.

In this case, you should store the IANA name of the time zone, not the Rails name, because the Rails name is not stable. It can be changed by the Rails team, and it has been changed in the past. The IANA name is stable, and it's the one that is used by the tzinfo gem, which is the one that Rails uses under the hood.

Lie #6: Time.parse is reliable


            Time.parse("2026-03-29 09:00:00")
            # => 2026-03-29 09:00:00 +0200

There is no +0200 in the input. The +0200 came from my laptop. If I deploy this code to a server in another zone, it returns a different value for the same string. This is a very common bug and the fix is quite easy as well.

Lie #7: 1:30 AM happens once

Daylight saving time was originally an industrial scheduling experiment. It became universal during World War One, then mostly stayed. Most countries that observe it move clocks by one hour, twice a year. And as you may know, each of those moves does something terrible to your code. 


            tz = TZInfo::Timezone.get("Europe/London")

            ghost = Time.new(2026, 3, 29, 1, 30, 0)

I don't know if you have seen a ghost before, but you're about to see one. 


            tz.local_to_utc(ghost)

That time never happened. Anywhere. To anyone. If you store it, you have stored a lie. 


            # => TZInfo::PeriodNotFound

And actually the code doesn't even work, you'll get a TZInfo::PeriodNotFound error. 


            tz = TZInfo::Timezone.get("Europe/London")

            ghost = Time.new(2026, 10, 25, 1, 30, 0)

I am generous, today'll see a second ghost, or actually something that looks like a ghost but is actually a clone. 


            tz.local_to_utc(ghost)

            # => TZInfo::AmbiguousTime

When the clocks go back, the same local time happens twice. If you try to convert it to UTC, you get an error because it's ambiguous. You have to choose which one you mean. 


            Time.zone.parse("2026-03-29 01:30:00")
            # => Sun, 29 Mar 2026 02:30:00 BST +01:00

            Time.zone.parse("2026-10-25 01:30:00")
            # => Sun, 25 Oct 2026 01:30:00 BST +01:00

That's not even the worst of it. If you use Time.zone.parse, it will pick one of the two times for you, without raising anything, hoping the right one is picked.

Truth #7

When parsing user input,
use TZInfo::Timezone#local_to_utc
and rescue both errors.

The fix to this problem is to use the timezone's local_to_utc method, which will raise an error if the time is invalid or ambiguous.

Okay, I know this is quite unlikely to happen, but keep in mind the time zone rules are changing all the time, you can never know what weird edge case your app will need to support in the future.

Lie #8: JSON and the database preserve your time


            Time.zone.parse("2026-06-25 09:00:00+01:00").to_json

At this point, you might be feeling safe, you're using zone, the string includes the offset, and you the time and date are pretty far to the scary ones we've seen so far. 


            Time.zone.parse("2026-06-25 09:00:00+01:00").to_json

            # => "2026-06-25T08:00:00.000Z"

And yes, it's true, nothing broke, nothing raised, this created a perfectly valid JSON string of a perfectly valid time.

But you lost a valuable information in the process, the offset. The time is now in UTC, and you have no way to know it was originally in BST.

Truth #8

Outside Ruby object, don't forget the zone.

Once you leave Ruby for serialization or storage, remember the zone won't follow you. If you need it, make sure to include it.

Database warning

PostgreSQL: timestamptz

MySQL: datetime

SQLite: no datetime type

If you need to store the time and keep the zone, you'll need to store them separately.

PotgreSQL has a timestamptz type that stores the time in UTC and the offset, but it doesn't store the IANA name of the time zone.

MySQL's datetime type doesn't store the time zone at all.

SQLite doesn't have a datetime type, so you have to store it as a string or an integer, and you have to handle the time zone yourself.

Lie #9: A minute is 60 seconds

TODO: picture of the Moon and the Earth

Earth's rotation is slowing down. The Moon pulls on our oceans, the friction acts like a brake, and over centuries the day gets longer. But do you know how it impacts you?

TAI

International Atomic Time

Today, the definition of the second is not based on the Earth's rotation anymore, but on the vibrations of a cesium atom. This is called TAI, or International Atomic Time, and it's a continuous time scale.

UTC

Coordinated Universal Time

But you and me, we use UTC, which is a time scale that is based on TAI but has leap seconds added to keep it in sync with the Earth's rotation.

Yes, you might have heard about leap seconds, but they exist. And we add or subtract them as needed, which is usually every few years. The last one was added in December 2016, and none have been added since.

TAI - UTC = 37 seconds

As of today, TAI is 37 seconds ahead of UTC.

This means, there are some cases where a minute is not 60 seconds.

UTC TAI UNIX timestamp
2016-12-31 23:59:59 2017-01-01 00:00:35 1483228799
2016-12-31 23:59:60 2017-01-01 00:00:36 1483228799
2017-01-01 00:00:00 2017-01-01 00:00:37 1483228800

There's a lot to say there. First, as we said, TAI is continuous, so it doesn't have a leap second.

UTC has a leap second, so it has a 23:59:60. This is a valid time in UTC.

UNIX timestamps count seconds since 1970-01-01 UTC but ignore leap seconds, so they don't represent them. This means that the same UNIX timestamp can correspond to two different UTC times, which is a problem if you're using UNIX timestamps to store time.

The 9 AM job.

  • Concrete example: user in Tokyo wants a daily notification at 9 AM their time
  • How do you store that?

          # ❌

  • Don't store a UTC instant

          # ✅
          t.string   :time_zone
          t.integer  :run_hour
          t.integer  :run_minute
          t.datetime :next_run_at
  • Store the intent: zone, hour, minute
  • Compute the instant when you need it
  • Recompute after every run

          Time.use_zone(user.time_zone) do
            target = Time.zone.now.change(
              hour: user.run_hour,
              min:  user.run_minute,
            )
            target += 1.day if target <= Time.zone.now
            target.utc
          end
  • Survives DST
  • Survives the user moving
  • Survives Paraguay
  • Intent stored, instant computed
